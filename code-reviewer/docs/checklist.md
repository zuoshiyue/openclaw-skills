# 代码审查检查清单

## 📋 使用说明

本清单基于 Clean Code、SOLID 原则、阿里巴巴开发规范等最佳实践，适用于人工代码审查和自动化检查。

---

## 🔴 严重问题（必须修复）

### 命名规范
- [ ] 类名使用大驼峰（UpperCamelCase）
- [ ] 方法名、变量名使用小驼峰（lowerCamelCase）
- [ ] 常量名全大写，单词间用下划线分隔
- [ ] 无拼音与英文混合命名
- [ ] 命名不以 `_` 或 `$` 开始/结束

### 代码质量
- [ ] 无魔法数字（未定义的常量）
- [ ] 无重复代码（DRY 原则）
- [ ] 无死代码（未使用的代码）
- [ ] 覆写 equals 必须同时覆写 hashCode

### 安全性
- [ ] 无硬编码密码、密钥
- [ ] 输入验证完整
- [ ] SQL 使用参数化查询（防注入）
- [ ] 无敏感信息日志输出

### 并发安全
- [ ] SimpleDateFormat 不定义为 static
- [ ] ThreadLocal 使用后有 remove()
- [ ] 单例模式线程安全
- [ ] 共享变量有适当的同步

---

## 🟡 警告问题（建议修复）

### 函数设计
- [ ] 函数参数不超过 3 个
- [ ] 函数只做一件事（单一职责）
- [ ] 函数名清晰表达功能
- [ ] 函数长度不超过 50 行
- [ ] 无布尔标志参数

### 代码结构
- [ ] 嵌套深度不超过 4 层
- [ ] 类长度不超过 500 行
- [ ] 文件长度不超过 1000 行
- [ ] 控制语句都有大括号

### 注释规范
- [ ] 公共方法有 JavaDoc
- [ ] 注释解释为什么而非做什么
- [ ] 无注释掉的代码
- [ ] TODO/FIXME 有明确标记

### 异常处理
- [ ] 不捕获 Exception 后不做处理
- [ ] 不抛出 generic Exception
- [ ] 有适当的日志记录
- [ ] 资源使用后正确关闭

---

## 🟢 最佳实践（推荐遵循）

### 命名
- [ ] 变量名可发音、有意义
- [ ] 同一概念使用相同词汇
- [ ] 避免单字母命名（循环变量除外）
- [ ] 抽象类以 Abstract/Base 开头
- [ ] 异常类以 Exception 结尾

### 设计原则
- [ ] 遵循单一职责原则（SRP）
- [ ] 遵循开闭原则（OCP）
- [ ] 遵循依赖倒置原则（DIP）
- [ ] 优先使用组合而非继承
- [ ] 面向接口编程

### 代码风格
- [ ] 使用 4 个空格缩进
- [ ] 单行不超过 120 字符
- [ ] 左大括号前不换行
- [ ] 优先使用 final 修饰符
- [ ] 类成员访问权限从严控制

### 性能优化
- [ ] 集合初始化指定容量
- [ ] 使用 entrySet 遍历 Map
- [ ] 避免循环内调用外层方法
- [ ] 字符串拼接使用 StringBuilder
- [ ] 优先使用基本类型而非包装类

### 测试
- [ ] 有单元测试覆盖
- [ ] 测试独立可运行
- [ ] 测试命名清晰（should_..._when_...）
- [ ] 包含边界条件测试
- [ ] 包含异常场景测试

## 📊 评分标准

### 分数计算
```
基础分：100 分

扣分项:
- 严重问题：-10 分/个
- 警告问题：-5 分/个
- 提示信息：-1 分/个

加分项:
- 注释率 15-30%: +5 分
- 测试覆盖率>80%: +5 分
- 无严重问题：+5 分
```

### 等级划分
| 分数 | 等级 | 说明 |
|------|------|------|
| 90-100 | ⭐⭐⭐⭐⭐ | 优秀，可直接合并 |
| 70-89 | ⭐⭐⭐⭐ | 良好，建议修复警告 |
| 50-69 | ⭐⭐⭐ | 一般，需要改进 |
| 30-49 | ⭐⭐ | 较差，建议重构 |
| 0-29 | ⭐ | 不可接受，重新编写 |

## 🎯 语言特定检查

### Java
- [ ] 使用@Override 注解
- [ ] 使用 Java 8+ 时间 API
- [ ] 使用 Optional 处理空值
- [ ] 使用 try-with-resources
- [ ] 泛型类型安全

### JavaScript/TypeScript
- [ ] 使用 const/let 而非 var
- [ ] 使用=== 而非==
- [ ] 使用箭头函数
- [ ] 使用解构赋值
- [ ] 使用 async/await 处理异步

### Python
- [ ] 遵循 PEP 8 规范
- [ ] 使用类型提示
- [ ] 使用上下文管理器
- [ ] 使用列表推导式
- [ ] 使用 f-string 格式化

## 📝 审查流程

### 1. 准备阶段
- [ ] 理解 PR/变更的目的
- [ ] 查看相关文档和需求
- [ ] 运行自动化检查工具

### 2. 审查阶段
- [ ] 使用本清单逐项检查
- [ ] 记录发现的问题
- [ ] 评估整体设计合理性

### 3. 反馈阶段
- [ ] 给出具体、可操作的反馈
- [ ] 区分必须修复和建议改进
- [ ] 提供改进建议或示例代码

### 4. 跟进阶段
- [ ] 跟踪问题修复情况
- [ ] 必要时进行二次审查
- [ ] 确认最终质量达标
